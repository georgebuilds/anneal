  H1 – default 1D is near real Metal ceiling (auto-vectorized / auto-cached)
  H2 – M3 f32 ALU peak is much lower than headline TFLOPs

Each thread: 4 independent f32 accumulators × 256 FMA iters = 2048 FMAs.

H2 FALSIFIED: M3 f32 ALU peak is NOT low — it is 8.2× the matmul rate.

──────────────────── Verdict ────────────────────────────────────────
H1 – CONFIRMED (with precision): The 83 GFLOP/s IS the real ceiling, but not
     because Metal "auto-vectorizes" in the code-optimization sense. The ceiling
     is determined by L2-cache bandwidth, not ALU.

H2 – FALSIFIED: The M3 GPU achieves 678 GFLOP/s on a pure-FMA workload. ALU
     is not the bottleneck.

H3 – FALSIFIED: Smem tiling, tile size, and upcast factor are all irrelevant.
     The kernel is not smem-access-latency limited.

──────────────────── Root cause: L2-bandwidth ceiling ───────────────
For 1024² f32 matrices (A=4MB, B=4MB):
  • The full A+B working set (8MB) fits in M3's ~12MB GPU L2 cache.
  • After warmup iterations, all loads hit L2 regardless of tiling strategy.
  • Logical L2 reads: 1024² outputs × 2×1024 loads × 4 bytes = 8.59 GB
  • At 25.75ms: effective L2 read bandwidth ≈ 334 GB/s.
  • "Threadgroup memory" (smem) on Apple's TBDR GPU is L2-backed — manual
    tiling changes WHERE the cache line sits (threadgroup vs implicit L1) but
    not WHICH bandwidth path is hit. Both paths use the same L2.
  • The default 1D kernel and every tiled variant all exercise ~334 GB/s of
    L2 bandwidth → same throughput.

Why 83 GFLOP/s and not 678?
  • The kernel is bandwidth-bound (not ALU-bound): each FMA requires loading
    2 f32 values from L2. At 334 GB/s L2 read bandwidth and 4 bytes/load,
    that is 334e9/8 = 41.75 billion loads/s. With 2 loads per FMA, the FMA
    rate ceiling is 41.75/2 = 20.875 billion FMAs/s ≈ 41.75 GFLOP/s.
    The 83 GFLOP/s (FMA counting = 2 FLOPs per FMA) = 41.75 Gfma/s. Consistent.

  • The 678 GFLOP/s FMA probe uses register-resident operands — no L2 reads
    in the hot loop. It proves the ALU has 678/83 ≈ 8.2× more headroom than
    the matmul exercises.

──────────────────── Recommendation ────────────────────────────────
To move beyond 83 GFLOP/s on this M3:
  1. MATRIX SIZE: Use N ≥ 8192 (A+B > 512MB) to exceed L2, making DRAM the
     bottleneck. Only then can smem tiling improve performance by reducing
     DRAM reads (metal threadgroup memory IS SRAM, below DRAM latency).
  2. FLOAT16: Halve bandwidth cost per element — could double effective throughput
     toward the f16 ceiling (tested separately in f16_test.go).
  3. DO NOT OPTIMIZE at 1024³: no dispatch/tiling/unrolling trick can move a
     L2-bandwidth-bound kernel. The kernel is at its natural ceiling for this
     working-set size.
  4. NEXT BENCHMARK TARGET: B4 — test at N=4096 or N=8192 to exit the L2-cached
     regime; re-run the same experiment grid to see if smem tiling starts
     helping when DRAM bandwidth is the limiting resource.
