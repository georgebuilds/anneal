FINDING: We are at the practical scalar-WGSL throughput ceiling (~85 GFLOP/s).
OptVectorize does not improve performance because:
  - WGSL's array<f32> does not guarantee 128-bit hardware loads for vec4 constructors.
  - Metal/Naga translates vec4<f32>(arr[i], arr[i+1], arr[i+2], arr[i+3]) to 4 scalar
    loads, not a single LDR Q instruction. The memory bandwidth is identical.
  - The (scalar * vec4) FMA expression generates 4 scalar FMAs, not a wider SIMD op.
  - The throughput ceiling is set by the Metal scalar f32 load+FMA rate (~85 GFLOP/s),
    which is unchanged regardless of vec4 packaging in the WGSL source.
This finding is consistent with the diagnostic result (test_output_diag.txt): every
configuration from 1024³ to 4096³ was already flat at ~85 GFLOP/s. The 512/1024³

BOUNDARY HANDLING: scalar epilogue per-component.
Each of the W=4 N-components in a vec4 accumulator is stored with an independent
bounds check: (Mexpr < Mreal) && (NexprBase + v < Nreal). When N is not divisible by
4, the out-of-bounds components are individually masked. Tests with N=17, N=30 confirm
correctness at max-abs-diff = 0. No padding or PADTO is used.

CONTIGUOUS-STRIDE PROOF — vec4 loads are genuine stride-1:
  Global B load (for each v in 0..3):
    data2[rowB * K_stride + (nWgID*NR*TS + nr*TS + lid.x*4 + v) * 1]
    → 4 consecutive N elements in the same row of B (row-major) ✓
  smB tile load (for each v in 0..3):
    sm159[nr*TS*TS + lid.y*TS + lid.x*4 + v]
    → 4 consecutive positions in smem ✓
  smB register load (inner-K at step k, for each v in 0..3):
    vec4<f32>(sm159[nr*TS*TS + k*TS + lid.x*4 + 0u], +1u, +2u, +3u)
    → 4 consecutive smem positions, stride-1 ✓
  NO strided gathers. The B column access (k varying, fixed col) is NEVER presented
  as a vec4 load — it stays scalar (smA is unchanged from B3).

WGSL FRAGMENT — inner-K block (first k=0 step, 64×64×64 kernel):
  @compute @workgroup_size(4, 16, 1)
  // workgroup_size.x = TS/W = 16/4 = 4 (vs 16 in B3 without vectorize)

  sm159[(0u + lid.y * 16u + lid.x * 4u + 1u)] = select(0.0, data2[rowB*K_stride + (colBase+1)*1], condB_1);
  sm159[(0u + lid.y * 16u + lid.x * 4u + 2u)] = select(0.0, data2[rowB*K_stride + (colBase+2)*1], condB_2);
  sm159[(0u + lid.y * 16u + lid.x * 4u + 3u)] = select(0.0, data2[rowB*K_stride + (colBase+3)*1], condB_3);

  // Inner-K: scalar A, vec4 B from contiguous smem, vec4 acc FMA
  let rA_0_0: f32    = sm158[0u + lid.y * 16u + 0u];           // scalar A
  let rBv_0_0: vec4<f32> = vec4<f32>(                          // contiguous B ✓

      sm159[0u + 0u + lid.x * 4u + 2u],

  acc0 = acc0 + (rA_0_0 * rBv_0_0);  // scalar * vec4 → vec4 accumulate
