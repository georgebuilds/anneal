──────────────────── Honest finding for 1024³ ────────────────────
The emitted shader is structurally exactly what B3 specifies (see WGSL fragment
below): MR + NR register pre-loads per inner-K step amortizing across MR*NR =
16 FMAs, MR*NR private accumulators, MR + NR cooperative global→smem tile
loads per outer-K step (coalesced + branchless). The arithmetic intensity at
the smem level rose from 0.5 FMA/load (B2 baseline) to 2.0 FMA/load (B3),
which should have helped if smem bandwidth were the bottleneck.

Best guess from inspecting the shader: at 1024³ the dispatch grid shrinks
from 4096 workgroups (B2 baseline OptTile) to 256 workgroups (B3, since each
WG now covers MR*TS × NR*TS = 64 × 64 outputs instead of 16 × 16). With
~10 GPU cores on the M3 and a workgroup_size of 256 threads, 256 WGs leave
only ~25 resident WGs per core — well below what's needed to hide memory
latency on Apple's architecture. So we traded a 16× reduction in smem
traffic (a non-bottleneck) for a 16× reduction in concurrency (the actual
bottleneck for this kernel size).

This is consistent with the B3 context note: "We are NOT bandwidth-bound …
We are occupancy/work-per-thread bound — each thread does one accumulate
per K step, no per-thread instruction-level parallelism, barriers dominate
per-thread useful work." Register blocking attacks per-thread ILP, but on
M3 the FMA throughput per thread already saturates at the baseline rate
(83 GFLOP/s ≈ same wall clock for 1× FMA/thread and 16× FMA/thread), so
the ILP gain doesn't move wall time. The remaining bottleneck is most
likely barrier/scheduling overhead in workgroup launch and warp issue,
NOT smem bank conflicts (the load addresses are uniform-broadcast within
a row for rA and within a column for rB, so banks are conflict-free).

Schedule-cache correctness re-verified by TestB3_ScheduleCache_HitCorrect:
two consecutive renders of the same b3Opts pipeline produce byte-identical
WGSL, so the kernel cache key continues to reflect the new AxisUpcast IR.

Value identity (max-abs-diff == 0) verified on:

  - matmul 17×32×32, 32×30×32, 17×30×35 (irregular — exercise upcast boundary
